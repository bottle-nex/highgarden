# Mock Polymarket & Server Polymarket Integration

Everything I added that touches Polymarket — both the mock app on one side and the server's Polymarket-consuming code on the other.

## The big picture

These two pieces exist to solve one problem: **your location can't reach real Polymarket, so the rest of the stack needs a stand-in.** Instead of stubbing API responses inside the server, I built a whole fake Polymarket (`apps/polymarket`) that speaks the same HTTP/WebSocket protocol. Then the server's Polymarket code (`apps/server/polymarket/*`, `apps/server/queue/*`, admin controllers) talks to it exactly like it will one day talk to the real thing. Flip `POLYMARKET_API_BASE` to `https://clob.polymarket.com` and zero server code changes.

The data flow end-to-end:

```
apps/polymarket                         apps/server
───────────────                         ───────────
BookSimulator ──► /gamma/markets  ◄──── GammaClient
              └─► /clob/book      ◄──── (future mirror service)
              └─► WS /ws          ◄──── (future mirror service)

                                        AutoLister ──► Prisma (Market + PolyMarket + Listing{PENDING})
                                        (runs on an interval,   triggers the admin approval queue
                                         also exposed as POST
                                         /admin/lister/run)

                                        Admin controllers ──► Prisma (flip Listing status to APPROVED/REJECTED)
```

## `apps/polymarket/` — the mock

A standalone Express + ws server on port 4000 that pretends to be Polymarket. Runs independently of the main server, seeds fake markets into memory, simulates an order book, and exposes the same REST + WebSocket shape as real Polymarket.

| ---------------- | --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `src/markets.ts` | `SeedMarket` interface + `SEED_MARKETS` const array (8 markets) | 8 hand-crafted fake markets: BTC $150k, ETH $5k, Fed cuts June, US recession, GPT-5 release, SpaceX Mars, Brazil WC, Apple car. Each has a realistic `question`, `description`, `endDate`, `volume24hr`, `liquidity`, and a YES/NO token pair with a starting price. | Without seed data the mock's `/gamma/markets` returns an empty list and the auto-lister has nothing to discover, so the admin queue stays empty forever. 8 is enough to exercise every code path (volume ordering, pagination, approve/reject, etc.) without cluttering the demo. The helper `mk(...)` keeps each market one-line so the data file stays readable. |

### Order book simulator

| File          | Class/exports                                                                                           | Purpose                                                                                                                                                                                                                                                                                                                                                                                                   | Why needed                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------- | ------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `src/book.ts` | `BookSimulator` class; `BookLevel`, `BookSnapshot`, `BookUpdate`, `BookEventType`, `BookListener` types | Per-token order book with 5 levels on each side. On `start()`, fires a tick every `intervalMs` (default 2s): for each token, does a random walk ±1 cent around the last trade price, rebuilds the 10 price levels with fresh sizes, and pushes an update to every listener. `getSnapshot(tokenId)` returns the current book. `subscribe(fn)` lets listeners register and returns an unsubscribe function. | Real Polymarket's book moves constantly. If our mock served static data, the mirror service and hedging bot wouldn't exercise the same timing races they'll face in production (stale quotes, price drift between quote-time and fill-time, partial fills when a level is exhausted). A random-walk simulator is the minimum viable version of "live data." The pub/sub `subscribe()` API exists so the WebSocket gateway can react to every tick without polling. |

### WebSocket handler

| File                                | Class/exports                                                                                                                                  | Purpose                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      | Why needed                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `src/gateways/market-ws.gateway.ts` | `MarketWsGateway` class, methods `attach(server, path)`, `close()`, internal `handleConnection`, `handleMessage`, `handleClose`, `formatEvent` | Attaches a `ws.WebSocketServer` to the http server at `/ws`. Each client gets a per-connection `subscriptions: Set<string>` of token IDs it cares about. When a client sends `{type:"subscribe", assets_ids:[...]}`, those tokens go into the set and the gateway immediately replies with a `subscribed` ack plus a current `book` snapshot for each. Every time the simulator ticks, the gateway pushes an event (`book`, `price_change`, or `last_trade_price`) to every client that subscribed to that token. Each event is shaped like `{channel:"market", event_type, asset_id, asks, bids, last_trade_price, timestamp}` — matching real Polymarket's `market` channel. On close, the gateway unsubscribes the listener and removes the client state. | Real Polymarket exposes a `market` WebSocket channel that streams live book updates. The mirror service (to be built) will subscribe via exactly this protocol. The gateway is split into its own class so bootstrapping it only takes one line (`wsGateway.attach(server, "/ws")`) and so the WebSocket logic is testable independent of the HTTP layer. The `WeakMap<WebSocket, SocketState>` avoids memory leaks if a client disconnects without the `close` handler firing. |

## `apps/server/` — the Polymarket-consuming code

The server-side code that talks to the mock (and will one day talk to real Polymarket). Three layers: HTTP client → background job → HTTP endpoints.

### HTTP client to Polymarket

| File                  | Class/exports                                                                                                                                                                             | Purpose                                                                                                                                                                                                                                                                                                                                                                                                                                 | Why needed                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `polymarket/gamma.ts` | `GammaClient` class with `fetchMarkets({limit, order, ascending})` instance method and `pickYesNoTokenIds(market)` static method; `GammaToken`, `GammaMarket`, `FetchMarketsParams` types | Typed HTTP client for Polymarket's Gamma API. Constructor takes a `baseUrl` (defaulting to `POLYMARKET_API_BASE` env var, falling back to `http://localhost:4000`). `fetchMarkets` builds the URL with query params, calls `fetch`, throws on non-2xx, casts the JSON to `GammaMarket[]`. `pickYesNoTokenIds` is a static helper that extracts the YES and NO `tokenId`s from a market's `tokens` array, throwing if either is missing. | The auto-lister needs _one_ typed surface to talk to Polymarket — scattering `fetch("/gamma/markets...")` calls around the codebase would rot fast. A class-per-API pattern means swapping `baseUrl` to real Polymarket is a one-line env change, and mocking in tests is a one-line `new GammaClient("http://test-mock")`. `pickYesNoTokenIds` is static because it's a pure function over the response shape — every consumer needs it, but it doesn't touch client state. |

### Background job

| File                              | Class/exports                                                                                                                                         | Purpose                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               | Why needed                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `queue/auto-lister.ts`            | `AutoLister` class with public `runOnce()`, `start()`, `stop()`; private `tick()`, `upsertOne(market)`; `AutoListerResult`, `AutoListerOptions` types | The heart of the queue. `runOnce()` fetches up to `batchLimit` markets from Gamma sorted by volume, then for each one: checks if a `Market` with that `polyMarketId` already exists (`findFirst`), skips if so, otherwise opens a Prisma `$transaction` that creates `PolyMarket` + `Market` + `Listing{status: PENDING}` atomically. Returns counts `{discovered, skippedExisting, failed}`. `start()` kicks off an immediate tick then sets a recurring interval. `tick()` is a re-entry-guarded wrapper around `runOnce()` — if the previous tick is still running (e.g. slow DB) the next tick is skipped instead of overlapping. | This is the queue's producer — without it, no markets ever appear in the admin panel. The re-entry guard is critical because `runOnce` can take longer than one `intervalMs` if the DB is slow, and concurrent ticks would race on the same `polyMarketId` uniqueness check and throw. The atomic transaction guarantees the admin never sees a half-created listing (e.g. a `Market` row without its `Listing`). The constructor accepts dependency injection (`{gamma, db, intervalMs, batchLimit}`) so tests can pass a stub `GammaClient` and an in-memory Prisma client without environment setup.                                     |
| `services/auto-lister.service.ts` | `export const autoLister = new AutoLister({intervalMs: Number(process.env.AUTO_LISTER_INTERVAL_MS ?? 60_000)})`                                       | Module-level singleton wrapping the `AutoLister` class, pre-configured from env.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      | Both `Services.boot()` (needs to call `.start()`) and the `RunListerController` (needs to call `.runOnce()` when an admin clicks "run now") must share the **same** instance. If they each `new AutoLister()`, `POST /admin/lister/run` would spin up a second timer and duplicate work. Reads `process.env.AUTO_LISTER_INTERVAL_MS` directly instead of going through the validated `ENV` export because this file is evaluated at module-load time, _before_ `Env.parse_env()` runs in `index.ts` — reading `ENV.X` would throw `Cannot read properties of undefined`. Same pattern `GammaClient` already uses for `POLYMARKET_API_BASE`. |

## How the pieces talk end-to-end

1. **Boot**: `apps/polymarket` starts → seeds 8 markets into the `BookSimulator` → starts the 2s tick loop → attaches the WS gateway.
2. **Boot**: `apps/server` parses env → `Services.boot()` connects Redis and calls `autoLister.start()` → `autoLister` does an immediate tick, hitting `http://localhost:4000/gamma/markets?limit=50&order=volume_24hr&ascending=false` via `GammaClient`.
3. **Discovery**: For each of the 8 returned markets, `AutoLister.upsertOne` checks `polyMarketId` uniqueness and, if new, writes `PolyMarket + Market + Listing{PENDING}` in one transaction.
4. **Admin**: You `GET /api/v1/admin/pending` → `ListPendingController` returns the 8 new listings.
5. **Admin action**: You `POST /api/v1/admin/approve/:marketId` or `/reject/:marketId` → the appropriate controller flips the listing status after checking it's still `PENDING`.
6. **Dedup**: On the next auto-lister tick (60s later, or on-demand via `POST /admin/lister/run`), the existing `Market` rows (including rejected ones) make the uniqueness check return `skipped`, so nothing duplicates.
7. **Future**: When the mirror service is built, it will subscribe to `ws://localhost:4000/ws` using the exact same protocol it'll later use against real Polymarket — and the `MarketWsGateway` will stream `book` / `price_change` events from the simulator.

That's the whole Polymarket scope across both apps.
